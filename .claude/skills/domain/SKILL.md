# OOP Java — 객체지향 설계 규칙

이 스킬은 **도메인 객체와 비즈니스 로직을 다룰 때**의 사고법을 정의한다.
"누가 이 책임을 가져야 하는가"가 모든 규칙의 핵심 질문이다.

구체적인 Before/After 예시는 같은 디렉토리의 `reference.md`를 참고할 것.
순수 Java 가독성 규칙(이름, 메서드 길이, early return 등)은 별도 스킬 `readable-java`에서 다룬다.

---

## 0. 적용 전 가드 — "이 코드가 비즈니스 규칙을 담고 있는가?"

`domain`/`service` 패키지 안에 있다고 해서 모든 코드가 도메인 영역인 건 아니다.
같은 패키지 안에도 단순 조회 메서드, 위임 코드, 인프라 어댑터가 섞일 수 있다.
**비즈니스 규칙이 없는 코드에 이 스킬의 규칙을 들이대면 오히려 나쁜 설계가 된다**
(예: 단순 위임 메서드에 Tell Don't Ask를 적용해 객체를 비틀어 짜는 식).

다음 질문을 통과해야 본문 규칙을 적용한다.

1. **이 클래스/메서드가 비즈니스 의사결정을 내리는가?**
   - 예: 할인 적용 여부, 주문 가능 여부, 상태 전이 가능 여부 → YES
   - 예: 데이터 전달, SQL 조립, HTTP 응답 매핑 → NO

2. **이 값이 "그냥 값"인가, "도메인 개념"인가?**
   - `(int rowIndex, int colIndex)` 두 개가 항상 같이 다니고 검증 규칙이 있다 → 도메인 개념(VO)
   - 그냥 페이지네이션용 `int page` → 그냥 값

3. **이 로직이 여러 곳에서 같은 형태로 반복되는가?**
   - 같은 비교/계산이 두 곳 이상에 있다 → 도메인 객체로 옮길 신호
   - 한 곳에서만 쓰이는 단순 변환 → 그대로 둬도 됨

가드를 통과하지 못한 코드는 본문 규칙을 적용하지 않고 `readable-java`만 따른다.

## 1. SRP — 하나의 클래스는 하나의 책임만

- 클래스의 변경 이유가 둘 이상이면 분리한다.
- "이 클래스의 역할을 한 문장으로 설명할 수 있는가?"를 자문한다. **"그리고"가 들어가면 분리 신호**다.
- "이 클래스는 보드를 관리하고 입력을 받고 승패를 판단한다" → 셋으로 나눠야 함.

판단법: 클래스 안의 메서드들을 그룹화했을 때 두 그룹 이상이 명확히 나뉘면, 그게 분리 라인이다.

---

## 2. Tell, Don't Ask — 가장 중요한 객체지향 원칙

객체의 상태를 꺼내서 외부에서 판단/조작하지 말고, **객체에게 시킨다**.

- 나쁨: `if (pass.getPassType() == FIXED && pass.getDuration() == lockerPass.getDuration())`
- 좋음: `if (pass.isSameDurationType(lockerPass))`

**리팩토링 신호: getter를 호출한 직후에 if 비교가 나오면 거의 99% Tell Don't Ask 위반이다.**

비교/판단/계산 로직이 호출자에 있으면, 그 로직이 여러 곳에서 중복되고 규칙 변경 시 모두 고쳐야 한다.
객체 안에 두면 한 번만 정의되고 호출자는 의도만 표현한다.

---

## 3. 도메인 객체로 책임 이동

비즈니스 규칙(할인 계산, 유효성 검증, 상태 전이 등)은 **그 규칙이 속한 도메인 객체 내부에** 있어야 한다.
서비스/Machine 클래스는 흐름만 조율하고, 결정은 도메인 객체가 내린다.

판단 기준:
- 서비스 메서드 안에 `getXxx()` 호출이 연속되고 그 결과로 if/계산이 이어진다 → 도메인으로 옮길 후보
- 같은 계산이 두 군데 이상 서비스에 있다 → 즉시 도메인으로 옮긴다

---

## 4. Value Object (VO)

의미 있는 값 묶음(좌표, 금액, 기간 등)은 원시 타입 그대로 들고 다니지 말고 VO로 감싼다.

VO 체크리스트:
- `private final` 필드 (불변)
- `private` 생성자 + 정적 팩토리 메서드 (`of`, `from`)
- 생성 시점에 유효성 검증 (잘못된 상태로는 객체가 만들어질 수 없도록)
- `equals` / `hashCode` 구현
- 상태를 바꾸는 메서드는 새 객체를 반환 (불변 유지)

VO를 만들지 말지의 신호:
- 두 개 이상의 원시 값이 항상 같이 다닌다 → VO 후보
- 그 값들에 대한 검증 규칙이 있다 → 거의 확정 VO
- 그 값 묶음에 대한 동작(이동, 비교)이 있다 → 확정 VO

## 5. 일급 컬렉션

`List<Pass>`를 그냥 들고 다니지 말고, 그 컬렉션을 감싸는 클래스를 만든다.

- 컬렉션 관련 동작(필터, 조회, 검증)을 그 클래스 안에 둔다.
- 이름은 복수형: `StudyCafeSeatPasses`, `Cells`, `Orders`.
- 외부에 컬렉션 자체를 노출하지 않는다 (Unmodifiable 또는 변환 후 반환).

일급 컬렉션으로 묶을 신호:
- `List<X>`에 대해 같은 stream 필터링이 호출부 곳곳에 반복된다
- 그 리스트에 대한 검증 규칙이 있다 (중복 없음, 최대 개수 등)
- 그 리스트 자체가 도메인 개념이다 (단순 조회 결과가 아니다)

## 6. Enum 활용 + 다형성으로 분기 제거

### 6.1 Enum의 기본 활용
- 분기 처리할 타입/상태 상수는 enum으로 만든다.
- "0: 진행 중, 1: 승리, -1: 패배" 같은 매직 넘버 + 주석 조합은 enum으로 즉시 바꾼다.
- 컴파일러가 모든 case를 강제하고 IDE 자동완성으로 후보가 다 보인다.

### 6.2 Enum에 행위를 둔다
- 타입에 따라 다르게 동작하는 로직은 `enum + 추상 메서드` 또는 `enum + 람다`로 표현하면 분기문이 사라진다.
- `if (type == A) ... else if (type == B) ...` 패턴이 보이면 각 타입이 자기 행동을 갖는 형태로 바꾼다.

### 6.3 다형성으로 if-else 제거 (인터페이스 + 구현체)
- enum이 부족할 만큼 복잡한 분기(상태가 많거나 행위가 무거움)는 인터페이스 + 구현체로 푼다.
- 새 타입이 추가될 때 기존 if-else를 손대지 않고 새 클래스만 추가하면 끝나는 구조를 지향.

---

## 7. 상속보다 조합

- 상속은 부모-자식이 **본질적으로 같은 종류**일 때만 사용한다. ("is-a" 관계가 명확해야 함)
- "is-a"가 애매하면 조합(has-a)을 쓴다.
- 공통 상태/행위는 별도 객체로 추출해서 필드로 들고 있는다.

신호:
- 부모 클래스에 `if (this instanceof X)` 같은 분기가 있다 → 상속이 잘못됨
- 자식 클래스가 부모의 일부 메서드만 의미가 있다 → 상속이 잘못됨
- 여러 클래스에 공통 상태가 있다 → 부모로 올리지 말고 별도 객체로 추출 후 조합

## 8. OCP — 확장에 열려있고 수정에 닫혀있다

새로운 종류(레벨, 결제 타입 등)가 추가될 때 기존 코드를 고치지 않고 새 클래스만 추가하는 구조를 지향한다.

- 인터페이스 + 구현체 패턴이 기본 도구.
- "또 다른 종류"가 나올 가능성이 있는 곳은 처음부터 인터페이스로 추상화.
- 단, **확장 가능성이 명백하지 않으면 미리 추상화하지 않는다** (YAGNI). 두 번째 종류가 등장할 때 추상화한다.

---

## 9. DIP — 추상에 의존하라

고수준 모듈이 저수준 구현 클래스에 직접 의존하지 않게 한다.

- 인터페이스를 두고 구현체는 외부에서 주입.
- 예: `SeatPassProvider` 인터페이스를 만들고, 파일 구현체/DB 구현체/API 구현체를 갈아끼울 수 있도록.
- 테스트할 때 가짜 구현체를 주입하기 쉬워진다는 부수 효과도 있다.

신호:
- `new XxxFileHandler()`처럼 클래스 안에서 구체 구현을 직접 생성하고 있다
- 외부 시스템(파일, DB, API)에 의존하는 클래스가 그 시스템의 구체 타입을 import한다

## 10. 클래스 내 멤버 선언 순서

`readable-java`의 2.6과 동일한 규칙을 도메인 객체에도 적용한다.

1. 상수(static final)
2. 인스턴스 필드 (모두 `private final`이 기본)
3. 생성자 (private)
4. 정적 팩토리 메서드 (`of`, `from`)
5. public 비즈니스 메서드 (도메인 행위)
6. private 메서드 (내부 헬퍼)
7. getter (최소한만 노출, 가능하면 두지 않는다)

도메인 객체에서는 **getter를 최소화**한다. getter가 많으면 Tell Don't Ask 위반의 통로가 된다.
정말 외부에서 값이 필요한 경우(View 렌더링, 영속화 등)에만 노출한다.

## 11. 작성 체크리스트

도메인 코드를 다 짠 후 점검한다.

- [ ] 클래스 책임이 "그리고" 없이 한 문장으로 설명되는가 (SRP)
- [ ] getter로 꺼낸 직후 if 비교를 하는 곳이 없는가 (Tell Don't Ask)
- [ ] 비즈니스 규칙이 서비스가 아닌 도메인 객체 안에 있는가
- [ ] 원시 타입 두 개 이상이 항상 같이 다니는데 VO로 묶이지 않은 게 있는가
- [ ] `List<X>`를 들고 다니며 외부에서 stream 필터를 반복하지 않는가 (일급 컬렉션)
- [ ] 매직 넘버 + 의미 주석 조합이 enum으로 대체되지 않은 채 남아있지 않은가
- [ ] 타입 분기 if-else가 다형성으로 풀릴 여지는 없는가
- [ ] 상속이 "is-a"가 아닌데 쓰이고 있지 않은가
- [ ] 클래스 내부에서 구체 구현체를 직접 생성하지 않는가 (DIP)
- [ ] 도메인 객체의 getter가 정말 필요한 것만 남아있는가
- [ ] 멤버 선언 순서가 [상수 → 필드 → 생성자 → 정적 팩토리 → public → private → getter] 인가

## 12. reference.md 사용법

각 규칙의 구체적인 Before → After 코드는 `reference.md`에 정리되어 있다.
규칙을 적용할지 헷갈리거나, 어떻게 고칠지 막막할 때:

1. SKILL.md에서 해당 규칙 번호를 확인 (예: `2 Tell, Don't Ask`)
2. reference.md에서 같은 번호 섹션을 찾는다
3. Before 코드의 어떤 부분이 문제였고, After에서 어떻게 풀었는지 비교
