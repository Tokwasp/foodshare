# Service 계층 작성 규칙

이 스킬은 **여러 도메인 객체·Repository·외부 연동을 한 트랜잭션 안에서 조율(orchestration)
하여 하나의 유스케이스를 완성**하는 코드를 다룰 때의 규칙을 정의한다.

핵심 질문은 단 하나다: **"이 서비스가 직접 _판단_ 하는가, 아니면 _흐름만 조율_ 하는가?"**
서비스 메서드 안에서 비즈니스 _판단_(할인 여부, 재고 충분 여부, 상태 전이 가능 여부)이
보이면, 그 판단은 도메인 객체로 내려보내야 할 코드다(→ oop-java).

> 코드 예제는 모두 같은 패키지의 **`reference.md`** 에 있다. 각 규칙 옆의
> "→ reference.md §N" 표시를 따라 해당 예제를 확인한다. 예제는
> `cafekiosk` 프로젝트의 실제 서비스 코드를 기준으로 한다.

## 0. 서비스의 책임 경계 — "흐름만 조율한다"

애플리케이션 서비스(`api/service` 패키지의 `@Service`)는 **유스케이스의 흐름을 조율**한다.
한 메서드는 보통 다음 흐름을 따른다.

1. Repository로 **필요한 도메인 객체를 조회**한다
2. 조회한 도메인 객체들에게 **일을 시킨다**(판단·계산·상태 변경은 도메인이 한다)
3. 변경된 객체를 **저장**한다
4. 결과를 **Response DTO로 변환해 반환**한다

### 0-1. 서비스가 _직접_ 하지 않는 것

- **비즈니스 판단/계산** — `if (stock.getQuantity() < n)` 같은 비교는 서비스의 일이 아니다.
  `stock.isQuantityLessThan(n)`처럼 **도메인 객체에게 묻는다**(→ oop-java §2 Tell Don't Ask).
- **상태 변경 로직** — 재고 차감·주문 상태 전이 등은 도메인 메서드(`stock.deductQuantity(n)`,
  `Order.create(...)`)가 수행한다. 서비스는 그 메서드를 호출만 한다.
- **HTTP/표현 관심사** — 서비스는 `ResponseEntity`, 상태 코드, `@RequestBody` 등을 모른다.
  그건 controller 스킬의 영역이다.

### 0-2. 판단 기준

> 서비스 메서드 본문에서 **`getXxx()` 호출 직후의 `if`/계산**, 또는 같은 비교가
> 두 곳 이상 반복되면, 그건 도메인 객체로 내려보낼 신호다(→ oop-java).
> 서비스에는 "조회 → 시킴 → 저장 → 변환"의 **흐름**만 남는 것이 이상적이다.

## 1. 클래스 구조

### 1-1. `@Service` + `@RequiredArgsConstructor` + 생성자 주입

→ reference.md §1

규칙:

- `@Service`를 붙인다.
- 의존(Repository, 다른 Service, Factory 등)은 **생성자 주입**으로 받는다.
  `@RequiredArgsConstructor`로 생성자를 생성한다.
- 필드는 `private final`로 선언한다. **필드 주입(`@Autowired` 필드)·세터 주입 금지.**

### 1-2. 서비스는 다른 서비스에 의존할 수 있다

→ reference.md §1-2

- 한 유스케이스가 다른 유스케이스(또는 공통 기능)를 필요로 하면, 그 서비스를 주입받아
  쓴다. 예: 통계 서비스가 메일 발송 서비스를 주입받아 사용.
- 단, 순환 의존(A→B, B→A)이 생기면 책임 분배가 잘못된 신호다. 공통 로직을 제3의
  컴포넌트로 추출한다.

### 1-3. 응집된 하위 책임은 협력 컴포넌트로 추출한다

→ reference.md §1-3

- 서비스 안의 한 덩어리 로직이 **자체 책임과 이름을 가질 만큼 응집**되어 있으면
  (예: "다음 상품 번호 생성"), 별도 클래스로 추출한다.
- 도메인 규칙이 아니라 **생성/조립 책임**이면 `@Component`(예: `XxxFactory`)로 둔다.
  (도메인 규칙이면 도메인 객체로 → oop-java)
- 추출 신호: 서비스 메서드 안에 "무엇을 만드는" 절차가 여러 줄 이어지고, 그 절차가
  서비스의 주 흐름과 결이 다르다.

## 2. 트랜잭션 경계

→ reference.md §2

- `@Transactional`은 **클래스 레벨**에 둔다(메서드마다 흩뿌리지 않는다).
- **조회 위주 서비스**는 클래스 레벨에 `@Transactional(readOnly = true)`를 기본으로 두고,
  **쓰기 메서드에만** `@Transactional`을 메서드 레벨로 재정의한다.
- 쓰기/읽기가 섞인 서비스에서 이 패턴을 쓰면 "기본은 읽기, 예외적으로 쓰기"가 코드로
  드러난다.

`readOnly = true`를 쓰는 이유:

- JPA가 **변경 감지(dirty checking)를 생략**해 약간의 성능 이점이 있고, 무엇보다
  **이 메서드는 쓰지 않는다는 의도**를 명시한다.
- CQRS 관점에서 읽기/쓰기 DB(또는 커넥션)를 분리할 때 `readOnly` 플래그가 라우팅
  기준이 된다.

> 트랜잭션 경계가 곧 **원자성의 경계**다. "조회 → 시킴 → 저장"이 한 메서드 안에서
> 모두 일어나야 부분 실패가 없다.

---

## 3. Response는 서비스가 완성한다

→ reference.md §3

- 서비스는 엔티티가 아니라 **`*Response` DTO를 완성해서 반환**한다. 엔티티를 서비스
  경계 밖으로 내보내지 않는다(지연로딩 예외·무한순환·민감필드 노출 방지).
- `*Response` 위치: `api/service/{도메인}/response` 패키지.
- 엔티티 → Response 변환은 `Response`의 **정적 팩토리 `of(entity)`** 로 캡슐화한다.
- 컬렉션 응답은 스트림으로 `Response::of` 매핑해 만든다.
- `*Response`는 **순수 데이터**만 담는다. `code/status/message` 같은 HTTP 메타는 넣지
  않는다(그건 controller의 `ApiResponse`/`ResponseEntity`가 씌운다 → controller 스킬).

---

## 4. 엔티티 생성·매핑의 위치

→ reference.md §4

- 엔티티 생성(`toEntity(...)` 등) 호출은 **서비스 안에서** 한다. Controller는 호출하지 않는다.
- 생성에 필요한 부가 값(예: 채번된 다음 상품 번호)이 있으면, 서비스가 그 값을 먼저
  구해서 인자로 넘긴다.
- 저장 후 `Response.of(savedEntity)`로 응답을 만들어 반환한다.

> 정리: **들어올 때** `요청 → 엔티티` (서비스), **나갈 때** `엔티티 → Response.of`
> (서비스). 엔티티는 서비스 안에서만 존재한다.

---

## 5. 비결정적 외부값은 파라미터로 주입받는다

→ reference.md §5

- `LocalDateTime.now()`, `UUID.randomUUID()`, 난수 등 **호출할 때마다 값이 달라지는
  것**을 서비스 메서드 _안에서_ 직접 호출하지 않는다. **파라미터로 받아온다.**
- 호출하는 쪽(Controller 등)이 `LocalDateTime.now()`를 만들어 넘긴다.

이렇게 하는 이유는 **테스트 결정성**이다:

- 서비스 안에서 `now()`를 부르면 테스트가 "지금 시각"에 의존해 검증이 어렵거나 깨진다.
- 시각을 파라미터로 받으면 테스트가 고정된 시각(`LocalDateTime.of(2025, 1, 1, ...)`)을
  넘겨 결과를 정확히 단정할 수 있다(→ service-test와 직접 연결되는 규칙).

> 신호: 서비스 메서드 본문에 `now()`/`random` 류 호출이 보이면, 그 값을 시그니처
> 파라미터로 끌어올린다.

---

## 6. 흐름은 의도가 드러나는 private 헬퍼로 분리한다

→ reference.md §6

- 조율 흐름이 길어지면, 단계별로 **의도를 드러내는 이름의 private 메서드**로 쪼갠다.
  public 메서드는 "무엇을 하는지"의 목차처럼 읽혀야 한다.
- 조회/생성 헬퍼는 의미가 드러나는 이름을 쓴다(예: `findProductsBy`, `createStockMapBy`).
- 인스턴스 상태(필드)를 쓰지 않는 순수 변환 헬퍼는 `private static`으로 둘 수 있다.
- 단, **메서드 이름·길이·분리 기준의 세부 규칙은 readable-java가 책임진다.** 이 스킬은
  "흐름을 단계로 분리한다"는 방향만 정한다.

---

## 7. 예외는 의미 있게 던진다

→ reference.md §7

- 실패 상황에서 **`null`을 반환하지 않는다.** 의미 있는 예외를 던진다.
  - 나쁨: 재고 부족 시 `return null` → 호출자가 분기.
  - 좋음: `throw new IllegalArgumentException("재고가 부족한 상품이 있습니다.")`
    (또는 더 구체적인 도메인/커스텀 예외).
- 예외 메시지는 명확한 한국어 문장으로 적는다(로그·응답에 그대로 노출될 수 있다).
- 서비스는 예외를 **던지기만** 한다. `try/catch`로 잡아 삼키지 않는다. 변환·응답 매핑은
  `@RestControllerAdvice`의 `ExceptionHandler`가 한 곳에서 처리한다(→ controller 스킬).

## 8. 흐름 요약

```
Controller            Service                         Domain / Repository
   │                     │                                    │
   ├────────────────────►│ (1) 조회   findXxxBy()  ───────────►│
   │                     │ (2) 시킴   domain.doSomething()    │  ← 판단·계산·상태변경
   │                     │ (3) 저장   repository.save()  ─────►│
   │                     │ (4) 변환   Response.of(entity)      │
   │◄────────────────────┤  *Response                         │
```

이 방향을 거스르는 코드(서비스가 직접 판단함, 서비스가 엔티티를 그대로 반환함,
서비스가 `now()`를 부름)가 보이면 잘못된 신호다.

## 9. 작성 체크리스트

서비스 계층 코드를 작성/수정한 뒤 점검한다.

### Service 클래스

- [ ] `@Service` + `@RequiredArgsConstructor`를 사용했는가?
- [ ] 의존을 `private final` 생성자 주입으로 받는가? (필드/세터 주입 없음)
- [ ] 메서드 본문이 "조회 → 시킴 → 저장 → 변환"의 흐름으로만 읽히는가?
- [ ] `getXxx()` 직후 `if`/계산이나, 반복되는 비교가 본문에 남아있지 않은가?
      (있으면 도메인 객체로 이동 → oop-java)
- [ ] 엔티티를 그대로 반환하지 않고 `*Response`로 변환해 반환하는가?
- [ ] 응집된 생성/조립 책임을 `@Component`(Factory 등)로 분리할 여지는 없는가?

### 트랜잭션

- [ ] `@Transactional`이 클래스 레벨에 있는가?
- [ ] 조회 위주라면 클래스 `@Transactional(readOnly = true)` + 쓰기 메서드만 `@Transactional`인가?

### 엔티티 / 응답

- [ ] 엔티티 생성(`toEntity(...)`)을 (Controller가 아니라) 서비스 안에서 호출하는가?
- [ ] `Response`를 서비스가 `Response.of(entity)`로 완성하는가? (HTTP 메타 미포함)

### 외부값 / 예외

- [ ] `now()`·난수 등 비결정적 값을 서비스 안에서 호출하지 않고 파라미터로 받는가?
- [ ] 실패 시 `null`을 반환하지 않고 의미 있는 예외를 던지는가?
- [ ] 서비스가 예외를 `try/catch`로 삼키지 않는가? (처리는 ExceptionHandler의 몫)

## 10. reference.md 사용법

각 규칙의 실제 코드 예제는 `reference.md`에 규칙 번호와 같은 번호로 정리되어 있다.
규칙 적용이 헷갈리거나 어떻게 짤지 막막할 때:

1. SKILL.md에서 해당 규칙 번호를 확인한다 (예: `2 트랜잭션 경계`)
2. reference.md에서 같은 번호 섹션의 코드를 본다
3. 흐름(조회→시킴→저장→변환)과 책임 분배가 어떻게 표현됐는지 확인한다
