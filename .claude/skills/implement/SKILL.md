# implement — 테스트·구현·보고 워크플로

도메인 파라미터 하나를 받아 한 기능을 **테스트 → 구현 → 보고** 3단계로 완성한다.
각 단계의 "어떻게"는 흩어져 있는 다른 스킬·문서에 있고, 이 스킬은 그 순서를
조율하고 단계 사이의 **검토 게이트**를 지키는 역할을 한다.

> **입력 파라미터: 도메인 이름 하나** (예: `chat`). 아래에서 `{도메인}`으로 표기한다.

---

## 0. 시작 전 점검

진행 전에 명세 파일이 있는지 먼저 확인한다.

1. `.claude/docs/{도메인}/docs.md`가 존재하는지 확인한다.
    - 이 파일은 **이 워크플로를 돌리기 전에** 사용자와 대화로 미리 작성해 둔
      아키텍처·구현 로직 명세다. 이 스킬은 그것을 **읽어서 따른다**(새로 만들지 않는다).
    - 파일이 없으면 진행을 멈추고, "먼저 `.claude/docs/{도메인}/docs.md`에 명세를
      정리하자"고 사용자에게 안내한다.
2. 명세가 있으면 1단계로 진행한다.

### 참고 경로 한눈에 보기

| 용도 | 경로 |
| --- | --- |
| 명세(입력) | `.claude/docs/{도메인}/docs.md` |
| 테스트 작성법(레이어별) | `.claude/skills/test/` |
| 구현 가이드(레이어별) | `.claude/skills/3layer/` |
| PR 형식 | `.claude/skills/pr-template/SKILL.md` |
| 테스트 보고(출력) | `.claude/docs/{도메인}/test.md` |
| 최종 보고(출력) | `.claude/docs/{도메인}/result.md` |

---

## 1. 테스트 작성 및 보고

명세를 보고 "무엇을 검증할지"를 정하고, 작성법을 보고 "어떻게 쓸지"를 따른다.

1. `.claude/docs/{도메인}/docs.md`를 읽는다 — 무엇을 만들지, 아키텍처, 구현 로직을 파악한다.
2. `.claude/skills/test/` 하위에서 작업할 레이어(controller / service / repository 등)에
   맞는 테스트 작성법 문서를 찾아 읽는다.
3. 명세 + 작성법에 맞춰 각 레이어의 테스트 코드를 작성한다.
    - 아직 구현이 없으므로 **테스트가 실패(red)하는 상태가 정상**이다.
4. 작성한 테스트를 요약해 `.claude/docs/{도메인}/test.md`로 보고한다.
    - 어떤 레이어에 어떤 테스트를 만들었고, 각 테스트가 검증하는 케이스가 무엇인지 적는다.
5. **여기서 멈춘다.** 사용자와 함께 테스트를 검토하고, 고칠 게 있으면 이 단계에서 수정한다.

> **검토 게이트 (반드시 지킨다):** 사용자가 "구현 들어가", "이대로 구현해줘" 같은
> 진행 지시를 하기 전에는 2단계로 넘어가지 않는다. 테스트가 만족스럽지 않으면
> 사용자 피드백을 받아 1단계 안에서 반복한다.

---

## 2. 구현

사용자 승인을 받은 뒤에만 시작한다.

1. `.claude/docs/{도메인}/docs.md`를 다시 참고한다(아키텍처·로직 기준).
2. `.claude/skills/3layer/` 하위에서 작업할 레이어에 맞는 구현 가이드를 찾아 읽고,
   각 레이어를 가이드에 맞게 구현한다.
    - 레이어별 세부 규칙(controller / service / persistence 등)은 해당 스킬을 따른다.
3. 1단계에서 작성한 테스트를 실행한다.
4. **모든 테스트가 통과(green)할 때까지** 구현을 수정하며 반복한다.
    - 세부 사항은 1단계(테스트) 검토에서 이미 합의됐으므로, 구현 중에는 멈춰서 다시 묻지 않고
      합의된 명세·테스트를 기준으로 구현을 끝까지 완료한다.
5. 테스트가 모두 통과하면 3단계로 넘어간다.

---

## 3. PR 형식 보고

구현 내용을 PR 본문 형식으로 정리해 파일로 남긴다.

1. `.claude/skills/pr-template/SKILL.md`를 읽고 그 형식과 절차를 따른다.
2. 구현·테스트 내용을 그 형식에 맞춰 채워 `.claude/docs/{도메인}/result.md`에 작성한다.
    - PR 성격(신규 기능 / 버그 수정 / 리팩터링 등)에 맞지 않는 섹션은 생략한다.
3. 보고 파일 경로와 핵심 내용을 사용자에게 알린다.

---

## 4. 작성 원칙

- **명세 우선.** `docs.md`에 없는 흐름·예외·동작을 추측해서 만들지 않는다. 단, 세부 논의는
  1단계(테스트) 검토에서 모두 끝내므로, 2단계 구현 중에는 멈춰 묻지 않고 합의된 명세·테스트대로
  구현을 완료한다.
- **게이트 준수.** 1단계 테스트 보고 후 반드시 사용자 검토를 기다린다. 한 번에 끝까지 달리지 않는다.
- **레이어 분리.** 테스트·구현 모두 레이어별 가이드(`.claude/skills/test/`, `.claude/skills/3layer/`)와
  레이어별 스킬을 따른다. 이 스킬은 순서와 게이트만 책임진다.
- **모든 출력은 `.claude/docs/{도메인}/` 아래.** 테스트 보고는 `test.md`, 최종 보고는 `result.md`.
