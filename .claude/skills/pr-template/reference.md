# PR 본문 작성 예시 모음

PR 성격별로 완성된 본문 예시를 담는다. SKILL.md의 템플릿을 실제로
어떻게 채우는지 보여주는 참고용이며, **이 문장들을 그대로 복사하지 말고**
실제 변경 내역에 맞춰 다시 작성한다.

## 목차
1. 신규 API 기능 (모든 섹션 사용)
2. 버그 수정 (엔드포인트 생략)
3. 리팩터링 (사용자 흐름·예외·엔드포인트 생략)
4. 설정·인프라 변경 (흐름 최소화)

---

## 1. 신규 API 기능 — 모든 섹션 사용

신규 엔드포인트가 추가되는 경우. 사용자 흐름, 서버 흐름, 예외 처리,
엔드포인트, 테스트를 모두 채운다.

````markdown
# 회원 가입 흐름
이메일 인증으로 발급된 토큰을 사용해 신규 회원을 등록하는 API 구현

## 사용자 흐름
```
이메일 인증 완료(토큰 발급)  →  회원 가입 요청  →  가입 완료(memberId 반환)
```

## 서버 흐름
```
입력 검증  →  중복 확인  →  인증 토큰 검증·소비  →  비밀번호 암호화  →  회원 저장
```
1. **입력 검증** (Bean Validation) — 이메일 형식, 비밀번호 정책(8–20자·영문 대소문자·특수문자), 닉네임 길이(2–10자), 주소 검증
2. **중복 확인** — 이메일·닉네임 중복 여부 조회
3. **인증 토큰 검증·소비** (Redis) — 토큰 일치 확인 후 삭제, 이미 소비됐으면 가입 실패
4. **비밀번호 암호화** (BCrypt) — 비밀번호를 암호화하여 저장
5. **회원 저장** — 회원 영속화 후 `memberId` 반환

---

## 예외 처리
- **입력값 검증 실패** (비밀번호 정책·닉네임 길이 등) — `MethodArgumentNotValidException` → **400**
- **이메일 중복** — `EmailDuplicatedException` → **409** · 이미 사용 중인 이메일입니다.
- **닉네임 중복** — `NicknameDuplicatedException` → **409** · 이미 사용 중인 닉네임입니다.
- **인증 토큰 없음/불일치** (미인증) — `EmailNotVerifiedException` → **400** · 이메일 인증이 완료되지 않았습니다.
- **인증 토큰 이미 소비됨** (동시 가입) — `EmailDuplicatedException` → **409** · 이미 사용 중인 이메일입니다.

## 엔드포인트
- **`POST /api/v1/members`** — 회원 가입 (이메일 인증 토큰 필요)

**Request**
```json
{
  "email": "test@example.com",
  "emailVerifyToken": "x3kPq...Zo",
  "password": "Passw0rd!@",
  "nickName": "닉네임",
  "address": {
    "roadAddress": "서울시 강남구 ...",
    "detailAddress": "101동 202호"
  }
}
```

**Response** `201 Created`
```json
{
  "code": 200,
  "data": { "memberId": 1 },
  "message": "회원 가입에 성공했습니다."
}
```

## 테스트
- ✅ **단위 테스트** (`MemberServiceTest`) — 가입 성공 / 이메일·닉네임 중복 / 토큰 만료·불일치 / 토큰 동시 소비(동시성) 케이스
- ✅ **timeout 테스트** (`MailServiceTimeoutTest`) — 응답 없는 SMTP 서버에서 5초 timeout 후 `MailSendException` 검증
- ✅ **Postman 수동 테스트** — 인증 → 가입 흐름 정상 동작 확인
````

## 2. 버그 수정 — 엔드포인트·Request/Response 생략

기존 동작의 버그를 고치는 경우. 서버 흐름은 "원인 → 해결" 관점으로
적고, 새 엔드포인트가 없으므로 엔드포인트 섹션은 생략한다.

````markdown
# 토큰 만료 시 500 응답 문제 수정
만료된 JWT 요청 시 401이 아닌 500을 반환하던 문제를 해결

## 서버 흐름
```
요청 수신  →  JWT 검증  →  만료 예외 처리  →  401 응답
```
1. **JWT 검증** (JwtAuthenticationFilter) — 토큰 서명·만료 검사
2. **만료 예외 처리** — 기존에는 `ExpiredJwtException`을 잡지 않아 필터 체인 밖으로 전파되며 500으로 처리됨
3. **401 응답** — 필터에서 `ExpiredJwtException`을 잡아 `AuthenticationEntryPoint`로 위임하도록 수정

## 예외 처리
- **토큰 만료** — `ExpiredJwtException` → **401** · 인증이 만료되었습니다. 다시 로그인해 주세요.
- **잘못된 토큰** — `MalformedJwtException` → **401** · 유효하지 않은 토큰입니다.

## 테스트
- ✅ **단위 테스트** (`JwtAuthenticationFilterTest`) — 만료 토큰 / 변조 토큰 / 정상 토큰 케이스에서 응답 코드 검증
- ✅ **수동 테스트** — 만료된 토큰으로 보호된 API 호출 시 401 반환 확인
````

## 3. 리팩터링 — 사용자 흐름·예외·엔드포인트 생략

동작 변경 없이 구조만 바꾸는 경우. 서버 흐름은 "변경 전후" 관점으로
적고, 외부 동작이 그대로이므로 사용자 흐름·예외·엔드포인트는 생략한다.

````markdown
# 주문 서비스 책임 분리
ProductService를 조회/변경 책임으로 분리

## 서버 흐름
```
변경 전: ProductService(조회 + 변경 혼재)
변경 후: ProductQueryService(조회)  +  ProductCommandService(변경)
```
1. **조회 책임 분리** — 상품 단건·목록 조회를 `ProductQueryService`로 이동
2. **변경 책임 분리** — 등록·수정·삭제를 `ProductCommandService`로 이동
3. **호출부 수정** — 컨트롤러가 용도에 맞는 서비스를 주입받도록 변경 (외부 API 시그니처 변화 없음)

## 테스트
- ✅ **단위 테스트** (`ProductQueryServiceTest`, `ProductCommandServiceTest`) — 분리된 서비스별 동작 검증
- ✅ **기존 테스트 통과** — 리팩터링 전 작성된 통합 테스트가 그대로 통과함을 확인
````

## 4. 설정·인프라 변경 — 흐름 최소화

빌드 설정, CI, 의존성 변경 등. 코드 흐름이랄 게 없으므로 제목·요약과
변경 내용 설명, 그리고 검증 방법만 적는다.

````markdown
# CI에 테스트 커버리지 측정 추가
GitHub Actions 파이프라인에 JaCoCo 커버리지 리포트 단계 추가

## 변경 내용
- `build.gradle`에 JaCoCo 플러그인 및 리포트 태스크 설정 추가
- PR 빌드 시 `jacocoTestReport` 실행, 커버리지 70% 미만이면 실패하도록 검증 단계 추가
- 리포트를 Actions 아티팩트로 업로드

## 테스트
- ✅ **CI 동작 확인** — 테스트 브랜치 PR에서 커버리지 리포트 생성 및 임계치 검증 정상 동작 확인
````

---

## 작성 시 참고

- 섹션 헤더(`## 사용자 흐름` 등)와 형식은 모든 PR에서 동일하게 유지하되, **해당 PR에 없는 섹션은 통째로 뺀다.**
- "변경 내용" 섹션은 정형 흐름이 없는 설정·인프라 PR에서만 사용하고, 코드 흐름이 있는 PR은 "서버 흐름"을 쓴다.
- 예시의 클래스명·메시지·경로는 모두 가상이다. 실제 코드의 이름과 문구를 그대로 옮긴다.
