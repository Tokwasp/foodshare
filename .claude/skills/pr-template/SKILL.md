# PR 본문 작성 규칙

이 문서는 일관된 Pull Request 본문을 작성하기 위한 형식을 정의한다.
사용자가 PR 작성을 요청하면, 아래 절차에 따라 변경 내역을 파악한 뒤
정해진 마크다운 템플릿을 채워서 제공한다.

> **언어 규칙: PR 본문은 한글로 작성한다.**
> (단, 코드 식별자, HTTP 메서드/상태 코드, 예외 클래스명, 기술 용어는 영문 그대로 사용한다.)

---

## 0. 작성 절차

PR 본문을 작성할 때는 다음 순서를 따른다.

1. 베이스 브랜치는 **`main`을 기본으로 한다.** (사용자가 다른 브랜치를 지정하면 그것을 따른다.)
2. `git log main..HEAD --oneline`으로 이 브랜치의 커밋 목록 확인
3. `git diff main...HEAD --stat`으로 변경된 파일 범위 파악
4. 필요 시 `git diff main...HEAD`로 핵심 변경 내용 확인
5. PR 성격(신규 기능 / 버그 수정 / 리팩터링 / 설정 변경)을 판단한다
6. **`references/reference.md`를 읽어** 해당 성격의 완성 예시를 확인하고, 어떤 섹션을 쓰고 뺄지 참고한다
7. 파악한 내용을 아래 템플릿의 각 섹션에 채운다
8. 해당 PR에 **없는 섹션은 통째로 생략**한다 (억지로 채우지 않는다)
9. 완성된 마크다운을 사용자에게 보여준다

> 변경 내역을 직접 확인할 수 없는 상황이면(예: diff 접근 불가),
> 사용자가 알려준 작업 설명만으로 템플릿을 채운다.

> **참고 파일:** PR 성격별로 모든 섹션을 채운 완성 예시는
> `references/reference.md`에 있다. 형식이 헷갈리거나 어떤 섹션을
> 넣을지 판단이 필요할 때 이 파일을 읽는다.

## 1. 전체 템플릿

아래 형식을 그대로 따른다. 적용되지 않는 섹션은 생략한다.

````markdown
# [작업/기능 제목]
[이 PR이 무엇을 하는지 한 문장 요약]

## 사용자 흐름
```
A 완료  →  B 요청  →  C 완료(반환값)
```

## 서버 흐름
```
단계1  →  단계2  →  단계3  →  단계4
```
1. **단계1** (기술/도구) — 무엇을 하는지 설명
2. **단계2** (기술/도구) — 무엇을 하는지 설명
3. **단계3** — 무엇을 하는지 설명

---

## 예외 처리
- **상황 설명** (부가 설명) — `ExceptionClass` → **HTTP상태** · 사용자 메시지
- **상황 설명** — `ExceptionClass` → **HTTP상태** · 사용자 메시지

## 엔드포인트
- **`METHOD /경로`** — 설명 (부가 조건)

**Request**
```json
{
  "field": "value"
}
```

**Response** `201 Created`
```json
{
  "code": 200,
  "data": { "id": 1 },
  "message": "성공 메시지"
}
```

## 테스트
- ✅ **단위 테스트** (`XxxTest`) — 어떤 케이스를 검증하는지
- ✅ **통합/timeout 테스트** (`YyyTest`) — 검증 내용
- ✅ **수동 테스트** — 어떤 흐름을 확인했는지
````

## 2. 섹션별 작성 가이드

### 제목 + 요약 (필수)

- `#` 한 개로 제목을 단다. 무엇을 만들었는지 명사형으로 짧게.
- 바로 아래 줄에 한 문장으로 핵심을 요약한다.

```markdown
# 회원 가입 흐름
이메일 인증으로 발급된 토큰을 사용해 신규 회원을 등록하는 API 구현
```

### 사용자 흐름 (선택)

사용자 관점에서 어떤 순서로 동작이 일어나는지 코드 블록 안에 `→`로 나열한다.
사용자 인터랙션이 뚜렷한 기능에서만 넣는다.

```
이메일 인증 완료(토큰 발급)  →  회원 가입 요청  →  가입 완료(memberId 반환)
```

### 서버 흐름 (선택)

먼저 코드 블록으로 처리 단계를 `→`로 한눈에 보여주고,
그 아래 번호 목록으로 각 단계를 설명한다.
**핵심 기술/도구는 괄호로 표기**한다(예: `(Redis)`, `(BCrypt)`, `(Bean Validation)`).

```
입력 검증  →  중복 확인  →  인증 토큰 검증·소비  →  비밀번호 암호화  →  회원 저장
```
1. **입력 검증** (Bean Validation) — 이메일 형식, 비밀번호 정책, 닉네임 길이 검증
2. **중복 확인** — 이메일·닉네임 중복 여부 조회
3. **인증 토큰 검증·소비** (Redis) — 토큰 일치 확인 후 삭제

### 예외 처리 (선택)

예외가 발생하는 상황을 한 줄씩 나열한다. 형식은 다음을 따른다.

```
- **상황 설명** (부가 설명) — `예외클래스` → **HTTP상태** · 사용자에게 보일 메시지
```

- 같은 예외가 다른 상황에서 재사용될 수 있으면 각 상황을 따로 적는다.
- 사용자 메시지가 없는 경우(검증 실패 등) `· 메시지` 부분은 생략 가능.

```markdown
- **이메일 중복** — `EmailDuplicatedException` → **409** · 이미 사용 중인 이메일입니다.
- **인증 토큰 없음/불일치** (미인증) — `EmailNotVerifiedException` → **400** · 이메일 인증이 완료되지 않았습니다.
```

### 엔드포인트 (선택)

API가 추가/변경되는 PR에서만 넣는다.

- `**`METHOD /경로`**` 형태로 엔드포인트를 적고 뒤에 설명을 단다.
- **Request**, **Response** 를 굵게 표기하고 JSON 예시를 코드 블록으로 넣는다.
- Response에는 상태 코드를 백틱으로 표기한다(예: `` `201 Created` ``).

### 테스트 (선택)

수행한 테스트를 체크 이모지(✅)로 나열한다.
테스트 종류를 굵게, 테스트 클래스명을 백틱으로 적고, 검증 케이스를 설명한다.

```markdown
- ✅ **단위 테스트** (`MemberServiceTest`) — 가입 성공 / 이메일·닉네임 중복 / 토큰 만료·불일치 케이스
- ✅ **timeout 테스트** (`MailServiceTimeoutTest`) — 5초 timeout 후 `MailSendException` 검증
- ✅ **Postman 수동 테스트** — 인증 → 가입 흐름 정상 동작 확인
```

---

## 3. 작성 원칙

- **변경된 내용만 적는다.** 실제 diff에 없는 흐름·예외·테스트를 추측해서 채우지 않는다.
- **섹션은 필요한 것만.** 리팩터링 PR이라면 엔드포인트·Request/Response가 없을 수 있다. 흐름과 변경 의도, 테스트만 적어도 된다.
- 흐름 다이어그램은 화살표 `→` 양옆에 공백 2칸을 둬서 가독성을 맞춘다.
- 사용자 메시지(예외 처리의 `·` 뒤)는 실제 코드의 문구와 일치시킨다.

---

## 4. 섹션 적용 가이드

| PR 성격 | 보통 들어가는 섹션 |
| ------ | ---------------- |
| 신규 API 기능 | 제목·요약 / 사용자 흐름 / 서버 흐름 / 예외 처리 / 엔드포인트 / 테스트 |
| 버그 수정 | 제목·요약 / 서버 흐름(원인·해결) / 예외 처리 / 테스트 |
| 리팩터링 | 제목·요약 / 서버 흐름(변경 전후) / 테스트 |
| 설정·인프라 변경 | 제목·요약 / 변경 내용 설명 / 테스트(있다면) |

각 성격별로 실제로 채운 완성 예시는 **`references/reference.md`**에 있다.
PR 성격을 판단한 뒤 해당 예시를 보고 형식을 맞춘다.
